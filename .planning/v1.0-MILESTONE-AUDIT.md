# v1.0 Milestone Audit — SmartDiscovery AI

**Audited:** 2026-05-27
**Status:** `gaps_found` — milestone NOT closeable

## Executive Summary

Phases 1–8 all completed plan execution and per-phase verification. However, a cross-phase integration check reveals that **E2E Flow 2 — the storefront visitor chat that IS the V1 core value statement** — is broken at three independent points. Anonymous-access edge protection (FND-03) is also unwired because the auth middleware file was named `proxy.ts` instead of `middleware.ts` (Next.js only auto-registers the latter).

The per-phase verification gates worked correctly for what they tested (each phase's internal contracts) but did not validate cross-phase reachability. This is the kind of drift `/gsd-audit-milestone` is designed to catch — and it caught it.

**V1 cannot ship until the four BLOCKER findings are closed.** All four are localized, well-scoped fixes (single-file edits in most cases).

## Findings

### BLOCKERS (4) — must be closed before v1.0 archive

#### B-1. `middleware.ts` missing — FND-03 unsatisfied

The project root has `proxy.ts` (exporting `function proxy(...)`). Next.js only auto-registers `middleware.ts`. Result: `/onboarding`, `/chat`, `/settings` have no edge-level auth guard. Anonymous direct hits load the embedded shell instead of redirecting to `/api/auth`.

**Fix:** Rename `proxy.ts` → `middleware.ts`. Change the exported function name to `middleware` (or use `export { proxy as middleware }`). Verify `config.matcher` is present and covers the three protected routes.

**Affected REQ:** FND-03.

---

#### B-2. Storefront bundle URL mismatch — STR-01 partial, STR-08 broken

`extensions/chat-drawer/assets/loader.js:45` fetches `/apps/smartdiscovery/storefront-manifest.json`. Backend serves `/apps/smartdiscovery/_meta/bundle-url` (file at `app/api/proxy/_meta/bundle-url/route.ts`). Mismatch returns 404; the storefront drawer JS bundle never loads.

**Fix:** Change loader.js to fetch `/apps/smartdiscovery/_meta/bundle-url` (the Liquid block comment already documents this URL correctly).

**Affected REQs:** STR-01, STR-08.

---

#### B-3. Storefront drawer is a placeholder — STR-05/06, SHR-04, IDN-04/05 broken

`extensions/chat-drawer/src/components/StorefrontDrawer.tsx:158-166` renders literal placeholder strings:
```tsx
<div>Chat coming up…</div>
<div>No history yet.</div>
<div>No saved products yet.</div>
```

The file header docstring claims composition of `ChatPane`, `HistoryPanel`, `SavedProductsPanel` from `lib/chat-ui`. But `grep -r '@/lib/chat-ui' extensions/` returns zero matches. Storefront visitors cannot interact with the chat at all.

**Fix:** Wire `StorefrontDrawer.tsx` to actually import + compose `ChatPane`, `HistoryPanel`, `SavedProductsPanel` with `StorefrontAdapter` + `useDbBackedHistoryStore` + `useDbBackedSavedProductsStore` hooks (these are already built and tested in `lib/chat-ui/`).

**Affected REQs:** STR-05, STR-06, SHR-04, IDN-04, IDN-05.

---

#### B-4. StorefrontAdapter endpoint hits wrong origin — STR-08 broken, EMB-07 partial

`lib/chat-ui/adapters/storefront.ts:6`:
```ts
readonly endpoint = '/api/proxy/chat';
```

From the merchant's storefront origin (their `.myshopify.com` domain), `fetch('/api/proxy/chat')` is same-origin to the storefront, NOT to the app backend. STR-08 explicitly requires App Proxy paths: `/apps/smartdiscovery/*`.

Worse, the test at `lib/chat-ui/__tests__/storefront-adapter.test.ts:16-18,108-117` actively asserts this incorrect value as "STR-08 compliance" — reinforcing the bug.

**Fix:** Change `endpoint` to `/apps/smartdiscovery/chat`. Update the test assertion + remove the misleading STR-08 comment. (The DB-backed stores already correctly use `/apps/smartdiscovery/conversations` and `/apps/smartdiscovery/saved-products` — the chat adapter is the lone outlier.)

**Affected REQs:** STR-08, EMB-07.

---

### WARNINGS (3) — non-blocking but worth fixing pre-launch

#### W-1. Failure-email retry deep link is inert — NOT-02 partial

`/onboarding/page.tsx` doesn't read `?retry=` query param. Functional retry works (operator clicks "Start sync" manually) but the deep link from the failure email lands them on the default view, not a "previous sync failed — retry?" affordance.

**Fix:** Add `useSearchParams` to onboarding page; when `?retry={syncRunId}` is present, render a "Your previous sync failed — Retry?" banner with a pre-confirmed button.

#### W-2. Admin /chat banner uses searchParams.shop while chat request uses session shop — ADM-05 partial

`/chat/page.tsx:16-19` reads `shop` from `searchParams ?? ''`. In a cross-shop edge case, banner could show `DEFAULT_MODEL.displayName` while the actual streamed request uses the session-bound shop's selection. T-04-25 documents this as accepted, but it's a visual inconsistency worth tightening.

**Fix:** Resolve `shop` from the embedded session token server-side in the page; `searchParams` should be a fallback / consistency check only.

#### W-3. Test pin reinforces B-4

`lib/chat-ui/__tests__/storefront-adapter.test.ts:16-18` and `108-117` assert `endpoint === '/api/proxy/chat'` and label this "STR-08 compliance". The assertion is wrong (it's the bug, not the contract). After fixing B-4, update this test to assert the App Proxy path.

## Verdict

**Status: `gaps_found`**

Per the workflow contract, this milestone CANNOT be archived via `/gsd-complete-milestone v1.0` until the four BLOCKERS are closed. The storefront chat — the V1 core value — does not work end-to-end despite passing per-phase verification.

Insert ONE phase that bundles all four blockers + three warnings + back-mark + Nyquist back-fill into a tight cleanup. Single PR back into `gsd/v1-phases-7-8` (or onto `main` after PR #2 merges).

### Path B — Accept and ship with caveats

Ship the existing PR #2 as a "backend-complete, storefront-pending" V1, and tackle the four blockers in a v1.1 hotfix immediately after. This is reasonable only if:
- The merchant beta launch can tolerate "admin playground works, storefront drawer does not"
- The team accepts shipping a marked-complete milestone whose core value statement is broken

Not recommended.

### Path C — Per-blocker insert

Four separate inserts (8.1, 8.2, 8.3, 8.4). More planning overhead than Path A; no real value because the blockers are tightly coupled to the same surface (storefront drawer).
