# Phase 4 — UI Design Contract

> Visual and interaction contract for the SearchService + Wire Chat refactor.
> Scope is narrow: a one-line preview-mode banner above the tab strip, refactored product-card rendering driven by Vercel AI SDK 6 `message.parts` tool-result entries, tool-call state affordances, and a locked no-results decision. This is **not** a greenfield design — it threads into the already-shipped chat surface and reuses every existing primitive.

## Accessibility Contract

Locked decisions (CONTEXT.md left this to discretion):

| Concern | Decision |
|---------|----------|
| Banner role | Plain `<div role="status" aria-live="off">` — not `aria-live="polite"`, because the banner is static at page load. The `role="status"` flags it as informational so AT users get context if they navigate into it. |
| Banner content reading order | Single text node. The display-name (`Gemini 2.5 Flash`) is wrapped in `<span class="text-foreground font-semibold">` so the model identity reads as emphasized text on screen readers but does NOT introduce a separate landmark. |
| Streaming assistant text | Wrap the assistant `<Response>` container in `<div aria-live="polite" aria-atomic="false">`. Atomic-false ensures only the *added* tokens are spoken, not the full message re-read on every chunk. |
| Tool-running pill | `<div role="status" aria-live="polite">Searching the catalog.</div>` — period instead of ellipsis so screen readers don't pause oddly. |
| Product result grid | `<ul role="list" aria-live="polite" aria-label="{N} matching products">`. The live announcement on count change tells AT users new results appeared. Re-announces ONLY when `output-available` first fires per tool call (not on every re-render). |
| ProductCard keyboard nav | Each card is a `<li>` containing the existing focusable controls (`Heart` button + "View" CTA). `Tab` order: Heart → View → next card's Heart. The card `<li>` itself is NOT `tabIndex={0}` — the controls inside it are the focus targets. |
| Zero-results & error affordances | `<div role="status">` with `aria-live="polite"`. Announced once per tool call. |
| Focus rings | All interactive elements use the existing `focus-visible:ring-ring focus-visible:ring-[3px]` pattern from `Button` and `TabsTrigger`. The new tool-status pill is NOT interactive (no `tabIndex`). |
| Color contrast | `text-muted-foreground` on `bg-muted/40` measured against `--background` in light mode: foreground `oklch(0.556 0 0)` ≈ #8B8B8B on background `oklch(1 0 0)` ≈ #FFFFFF yields ~4.6:1 — passes WCAG AA for small text (4.5:1 required). Confirm before Phase 4 verification gate. |
| Dark mode | The chat surface inherits `.dark` class behavior from existing tokens. No new dark-mode work in Phase 4 — verify the banner uses `bg-muted/40` (which adapts) rather than a hardcoded color. |

## Risks and FLAGs for ui-checker

These are honest spots the checker should weigh harder than auto-pass:

1. **Banner height is 28px** (`py-1.5` + `text-xs`). Below the 32px "comfortable scan-line" baseline. Justified because it is a passive mode indicator with no interactive zone — but if the checker's spacing dimension is strict about minimums regardless of interactivity, this needs a deliberate PASS rationale.
2. **ProductCard uses inline hex `#008060`, `#e1e3e5`, `#f6f6f7`, `#202223`, `#6d7175`** — these are Shopify Polaris palette literals that pre-date the shadcn `new-york` token migration. Phase 4 does NOT refactor them (out of scope). The accent-reserved list above explicitly carves out the `#008060` usage. Flag for Phase 5 (`lib/chat-ui/` extraction) to harmonize with `--accent` / `--primary` tokens.
3. **Greeting text apostrophes** use HTML entity `&apos;` in the existing JSX. The locked copy in this spec uses straight ASCII `'` in markdown but executor must render via `&apos;` to match the existing JSX style in `chat.tsx`. Documented but checker should not flag the markdown straight-quote.
4. **No primary CTA in this phase.** Empty CTA row in copywriting is intentional, not a gap. Checker copywriting dimension should accept "n/a — phase is a refactor, not a new feature."
5. **No destructive actions in this phase.** Empty destructive row is intentional. Same justification.
6. **Tool-error affordance is intentionally quiet** (muted text + 12px icon). If checker grades "errors must be visually loud," this needs an exception note — the CONTEXT.md side-channel design says the model absorbs the error in text, so the UI is a polite secondary signal, not the primary one.
7. **Embedded admin host injects Polaris JS via `<Script>`** but we render shadcn primitives. There is no visual conflict in the chat surface itself (the iframe is just the host shell), but the executor must not import Polaris React components.
8. **Vercel AI SDK 6 tool-state field rename** — RESEARCH.md L11 flags `inputSchema` vs `parameters` as a v5→v6 surface. UI-SPEC consumes `part.state` and `part.input` / `part.output` which are stable v6 names. No risk to this spec, but executor should consult RESEARCH.md §"Concrete Syntax" before typing the renderer.
