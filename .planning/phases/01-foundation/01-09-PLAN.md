<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Manual smoke verification (embedded admin iframe)</name>
  <what-built>
    Phase 1 has rewritten the schema, re-enabled the proxy auth guard, deleted token-leaking console.logs, and added the shared `withShopifySession` wrapper. The automated gates in Task 1 cover the structural invariants. This human checkpoint covers the things automation can't: live OAuth roundtrip, embedded iframe behaviour, and absence of stdout token leaks under a real `bun dev` session.
  </what-built>
  <how-to-verify>
    Step 1 — Clean dev DB and start the app. Run:
      bunx prisma migrate reset --force
      bunx prisma migrate deploy
      bunx prisma generate
      bun dev
    Confirm `bun dev` boots without an error mentioning `runtime` (Pitfall 1 would surface here if proxy.ts still exported runtime).

    Step 2 — Install on a Shopify dev store (use the partner-dashboard test store referenced in shopify.app.toml). Trigger the OAuth flow by visiting the install URL. Watch the terminal running `bun dev` while completing OAuth. Search stdout/stderr (grep the captured output or eyeball it) for any line that contains "Bearer " or a JWT-shaped string (three base64 segments separated by `.`). Expected: NO matches. This validates FND-02 in the real run, not just by grep on the source.

    Step 3 — In the dev store admin, open the SmartDiscovery AI app from the sidebar (iframe loads `/onboarding?shop=<store>.myshopify.com`). Expected: page loads, no redirect loop, the "Start sync" button is visible. This validates FND-03 (proxy allows authenticated traffic through).

    Step 4 — In a separate browser tab (not the iframe — direct navigation), visit `http://localhost:<port>/onboarding` (NO `?shop=` param). Expected: 307 redirect to `/api/auth`. Then visit `http://localhost:<port>/onboarding?shop=nonexistent-store.myshopify.com`. Expected: 307 redirect to `/api/auth?shop=nonexistent-store.myshopify.com`. Both validate FND-03 (unauthenticated/unknown-shop access is blocked at the page level).

    Step 5 — From the onboarding page in the iframe, click "Start sync". Expected: the request returns 200 (or 401 with a clearly enumerated error code). Either response is acceptable — this validates that the new `withShopifySession` wrapper is wired correctly. A 500 or an unhandled error here is a failure.

    Step 6 — Inspect the live DB (using `bunx prisma studio` or `psql`) to confirm:
      a. `products`, `product_variants`, `product_images`, `product_options`, `product_embeddings` tables exist with a `shop` column NOT NULL.
      b. `shopify_sessions` table is intact with at least one row (the dev-store install session from Step 2).
      c. The pgvector extension is enabled.
    This validates FND-01 against the live DB, not just the schema file.
  </how-to-verify>
  <resume-signal>Type "phase 1 verified" when steps 1-6 all pass, OR describe the failure (which step, what error message).</resume-signal>
  <acceptance_criteria>

    - OAuth flow completes without any Bearer/token strings appearing in stdout.
    - Embedded admin loads `/onboarding?shop=…` in the iframe with no redirect loop.
    - Direct navigation to `/onboarding` (no shop) → 307 to `/api/auth`.
    - Direct navigation to `/onboarding?shop=<unknown>` → 307 to `/api/auth?shop=<unknown>`.
    - "Start sync" returns a well-formed response (200 or 401 with enumerated code), not a 500.
    - Live DB inspection confirms `shop NOT NULL` on 5 product tables and shopify_sessions intact.
  </acceptance_criteria>
