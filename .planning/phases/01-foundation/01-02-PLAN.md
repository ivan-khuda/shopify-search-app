      3. If `!payload.dest`, throw `new ShopifyAuthError('invalid_dest')` (NOT 'invalid_token' — this is the D-06 split).
      4. Inside try/catch, parse `new URL(payload.dest).hostname` into `shop`. On catch throw `new ShopifyAuthError('invalid_dest')`.
      5. If `!shop.endsWith('.myshopify.com')`, throw `new ShopifyAuthError('invalid_shop_domain')`.
      6. Compute `sessionId = shopifyClient.session.getOfflineId(shop)`; call `await sessionStorage.loadSession(sessionId)`. If the result is `null` or `undefined`, throw `new ShopifyAuthError('no_offline_session')`.

    - `withShopifySession(handler: (ctx: { shop: string; session: Session; req: Request }) => Promise<Response>): (req: Request) => Promise<Response>` calls `verifyShopSessionToken(req)` inside try/catch:

      - On caught `ShopifyAuthError`, returns `NextResponse.json({ error: err.code }, { status: err.status })`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client (Shopify Admin iframe) → API route | Untrusted: client supplies a Bearer JWT in `Authorization`. Verification happens in `verifyShopSessionToken`. |
| API route → SDK (`@shopify/shopify-api`) | Trusted: SDK is the canonical JWT verifier (HS256 + 10s clock skew + audience check per Q3). |
| API route → Database (offline session storage) | Trusted (intra-app). `sessionStorage.loadSession` returns `Session | undefined`; missing session → reject. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Spoofing (session token replay) | `verifyShopSessionToken` decode step | mitigate | Delegate JWT verification to `shopifyClient.session.decodeSessionToken` which enforces HS256 signature, 10s clock-skew tolerance, and audience check (Q3). No custom JWT parsing. |
| T-1-04 | Spoofing (forged shop in token) | `verifyShopSessionToken` shop-derivation step | mitigate | Derive shop ONLY from verified `payload.dest`; assert `hostname.endsWith('.myshopify.com')` before loading offline session. Client-supplied shop param never reaches this code path. |
| T-1-05 | Information Disclosure (verbose error leaks) | `ShopifyAuthError` message | mitigate | The error message is a fixed string `Shopify auth error: <code>` — no token, no shop, no header values included. The wrapped HTTP response body is `{ error: <code> }` — no stack, no secrets. |
| T-1-03 | Information Disclosure (logging) | `lib/shopify/auth.ts` | mitigate | Per D-10, NO `console.log` is introduced. Grep gate in acceptance criteria enforces this. |
</threat_model>
