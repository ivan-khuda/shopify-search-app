# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Storefront Surface.
> This phase introduces two net-new UI surfaces that do NOT exist anywhere in the codebase today:
> (1) a floating action button (FAB) injected into merchant storefronts via a Shopify Theme App Extension,
> and (2) a side drawer shell that hosts the `lib/chat-ui/` chat components via `StorefrontAdapter`.
> All shared chat components (`ChatPane`, `ChatMessage`, `ProductCard`, `HistoryPanel`, `SavedProductsPanel`)
> are REUSED unchanged from `lib/chat-ui/`. This spec covers the drawer shell, FAB, and identity
> onboarding affordances ONLY — do not re-design the chat surface itself.
>
> Generated by gsd-ui-researcher (2026-05-26).

**Accent reserved for (10% rule):**
1. FAB background circle (`bg-[var(--sd-accent,#008060)]`).
2. FAB focus ring (`ring-[var(--sd-accent,#008060)]`).
3. Active tab trigger text color (`data-[state=active]:text-[var(--sd-accent,#008060)]`).
4. Sparkles brand-mark tile in drawer header (`bg-[#008060]` — fixed, not merchant-overridable, matches admin surface identity).
5. ProductCard "View" CTA text (`text-[#008060]`) — carried from Phase 5, no change.

No other element may use the accent color. Prompt chips use `bg-muted` (secondary), not accent.

**Z-index strategy (STR-07):**

| Layer | z-index | Element |
|-------|---------|---------|
| Theme content | 0–99 | All merchant theme elements |
| Dawn app blocks | 100–199 | Dawn theme UI (observed max ~100) |
| Sense / Craft overlays | 200–299 | Known overlay elements in these themes |
| SD Drawer scrim | 2000 | `<div class="sd-scrim">` full-viewport overlay |
| SD Drawer panel | 2001 | `<aside class="sd-drawer">` sliding panel |
| SD FAB | 2002 | `<button class="sd-fab">` floating button |

Rationale: z-index 2000+ is safely above Dawn's documented max of ~400, Sense's ~300, and Craft's ~200. Using a 4-digit z-index also avoids conflicts with common merchant app installs (chat widgets, cookie banners typically land at 999–1999).

Implementation: z-index values are applied inline on the custom element's shadow DOM root host element, or as CSS variables on the `:host` selector if shadow DOM is used. If light DOM is used (simpler for Tailwind), use `!important` on the `z-index` declarations to prevent theme cascade override.

## Theme Isolation Strategy

The extension bundle runs in the merchant's theme context. The drawer must render correctly regardless of what CSS the theme applies.

**Approach: Scoped custom element with self-contained styles.**

- Wrap FAB + drawer in a single `<smartdiscovery-app>` custom element.
- The extension's esbuild/Vite output generates a self-contained CSS file that redeclares all required CSS variables as inline `:root` equivalents scoped under `.sd-root` (a class applied to the custom element's container div).
- This avoids shadow DOM (which would break Tailwind utility classes) while still providing sufficient isolation from theme resets.
- Required CSS variable declarations injected by the extension's `<style>` block:

```css
.sd-root {
  --background: oklch(1 0 0);
  --foreground: oklch(0.145 0 0);
  --muted: oklch(0.97 0 0);
  --muted-foreground: oklch(0.556 0 0);
  --border: oklch(0.922 0 0);
  --destructive: oklch(0.577 0.245 27.325);
  --radius: 0.625rem;
  /* Merchant override — set by App Embed block settings */
  --sd-accent: #008060;
  /* Font reset to prevent theme leakage */
  font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif;
  font-size: 16px;
  line-height: 1.5;
  color: var(--foreground);
  box-sizing: border-box;
}
.sd-root *, .sd-root *::before, .sd-root *::after {
  box-sizing: inherit;
}
```

- The extension does NOT import `app/globals.css`. It imports its own CSS entry that includes these variable declarations plus the compiled Tailwind utility classes scoped under `.sd-root`.
- `next/image` is NOT used in the extension bundle. ProductCard image rendering in the storefront context uses a standard `<img>` tag with explicit `width`/`height` attributes (or `loading="lazy"`), since the Next.js Image Optimization API is not accessible from the extension's vanilla JS context.

## Drawer Open/Close Animation

| Trigger | Animation | Duration | Easing |
|---------|-----------|----------|--------|
| Open (desktop) | Slide in from right (`translateX(100%)` → `translateX(0)`) | 250ms | `ease-out` |
| Close (desktop) | Slide out to right (`translateX(0)` → `translateX(100%)`) | 200ms | `ease-in` |
| Open (mobile) | Slide up from bottom (`translateY(100%)` → `translateY(0)`) | 250ms | `ease-out` |
| Close (mobile) | Slide down (`translateY(0)` → `translateY(100%)`) | 200ms | `ease-in` |
| Scrim (mobile) | Fade in/out (`opacity: 0` → `opacity: 1`) | 200ms | `ease-out` |
| FAB icon swap | Opacity cross-fade (`duration-150`) | 150ms | `ease-in-out` |

Implementation: use `motion` (already in `package.json`) `AnimatePresence` + `motion.aside`. Keep total motion budget under 250ms. No spring animations on the drawer shell (reserved for within-tab transitions in future phases).

**Theme Editor guard:** When `window.Shopify?.designMode === true`, mount the FAB but set `initialOpen = false` and do not auto-open the drawer. This satisfies STR-07.

## Interaction Contract

| Interaction | Behavior |
|-------------|----------|
| FAB click (closed) | Open drawer, change FAB icon to `X`, set `aria-expanded="true"` |
| FAB click (open) | Close drawer, revert FAB icon to `Sparkles`, set `aria-expanded="false"` |
| Scrim click (mobile only) | Close drawer |

| Chip click | Pre-fill ChatPane textarea + submit immediately (no extra confirm step) |
| Tab switch | Standard shadcn `Tabs` — click tab label to switch content panel |

| "Clear All" (History) | Clears localStorage-backed history store — no confirmation (same as admin) |
| Page reload | Drawer state resets to closed. `visitor_id` persists in localStorage. Conversation history and saved products persist via store hooks. |

## Identity Onboarding (Transparent)

Per IDN-01 and IDN-02: identity setup has NO visible UI affordance in Phase 6.

| Trigger | What happens | What the visitor sees |
|---------|-------------|----------------------|
| First drawer open | `StorefrontAdapter.getRequestBody()` generates `crypto.randomUUID()`, writes to `localStorage['smartdiscovery.visitor_id']` | Nothing (transparent) |
| `window.Shopify.customer` present | Adapter reads `customer.id`, sends alongside `visitor_id` on each request; server merges records | Nothing (transparent) |
| Subsequent sessions | `localStorage` key present; existing `visitor_id` returned | Conversation history and saved items appear in drawer from prior session |

No "sign in to sync" prompt, no cookie banner, no opt-in flow. This is V1 scope. Visibility of merged history (IDN-04) is the only user-facing identity signal — the data just appears.

## Risks and FLAGs for ui-checker

1. **`next/image` cannot be used in the extension bundle.** The extension is a standalone JS bundle injected into the merchant's theme — it is not served by the Next.js runtime. `ProductCard` uses `next/image` from `lib/chat-ui/`. The storefront extension must either (a) wrap `ProductCard` with an image adapter that swaps `next/image` for a plain `<img>` tag, or (b) maintain a separate `StorefrontProductCard` component. Flag for executor: this is the one shared component that requires a surface-specific adaptation beyond the adapter interface.

2. **Tailwind v4 utility class scoping.** The extension's Tailwind build must include only the classes used by the drawer components. A full Tailwind purge scan across `lib/chat-ui/` is required during the extension build configuration step. Executor must ensure the extension's `postcss` config targets the correct source files.

3. **FAB position when merchant installs competing chat apps.** Dawn, Sense, and Craft themes don't reserve the bottom-right corner. However, Tidio, Gorgias, and similar chat apps conventionally occupy `bottom-6 right-6`. If a merchant has both installed, the FABs will visually overlap. This is out of scope for V1 (no z-index negotiation protocol) — document the known limitation in a code comment on the FAB position classes.

4. **Mobile bottom-sheet height `85vh`.** On short iOS devices (iPhone SE) with Safari's dynamic toolbar, 85vh may produce scroll issues. Executor should test on 375×667 viewport. Fallback: `h-[min(85vh,600px)]` if the vh causes cut-off.

5. **`visitor_id` localStorage key.** `StorefrontAdapter` already uses `'smartdiscovery.visitor_id'`. The Phase 6 extension MUST use this exact same key — do not introduce a second key. The `StorefrontAdapter` is the canonical source of the key constant.

6. **`Shopify.designMode` timing.** `window.Shopify?.designMode` may not be set synchronously at script load time in all Theme Editor contexts. The guard should check on first FAB click, not at mount. Drawer auto-open is controlled by the initial `isOpen` state which defaults to `false` — the guard only needs to prevent programmatic open calls, not react to `designMode` reactively.

7. **App Proxy HMAC verification** is a backend concern, not a UI concern. This spec covers the frontend boundary only. The executor must cross-reference the API spec for `/api/proxy/chat` HMAC implementation.
