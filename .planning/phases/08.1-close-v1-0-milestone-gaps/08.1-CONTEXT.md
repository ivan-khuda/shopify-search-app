### B-3: StorefrontDrawer composition

- **D-08:** **Direct composition in `StorefrontDrawer.tsx`** — replace the placeholder JSX (lines ~158-166) with imports of `ChatPane`, `HistoryPanel`, `SavedProductsPanel` from `@/lib/chat-ui`, plus `StorefrontAdapter` from `@/lib/chat-ui/adapters/storefront`, plus `useDbBackedHistoryStore` + `useDbBackedSavedProductsStore` hooks. No new shared shell wrapper — this is the right level of abstraction. Admin chat-shell does the same.
- **D-09:** **Tab routing pattern follows admin chat-shell.tsx.** Use the existing shadcn `Tabs` primitive — the storefront drawer already imports it. Mount `<ChatPane adapter={...} />` in the Chat tab, `<HistoryPanel ... />` in the History tab, `<SavedProductsPanel ... />` in the Saved tab. Pass `shop` + `visitorId` to the stores via the existing visitor-identity flow.
- **D-10:** **Minimal RED test.** Add ONE component test at `extensions/chat-drawer/src/components/__tests__/StorefrontDrawer.test.tsx` that renders the drawer (mock store hooks + adapter) and asserts each of `ChatPane`, `HistoryPanel`, `SavedProductsPanel` is in the DOM. The placeholder-text smell is what slipped past per-phase verification — the new test prevents regression. Do NOT replicate the per-component RTL coverage that already lives in `lib/chat-ui/__tests__/`.

### W-1: Onboarding retry deep link

- **D-11:** **`/onboarding/page.tsx` reads `?retry={syncRunId}` via `useSearchParams`.** When the param is present:
  - Lookup the SyncRun via `/api/shopify/sync/status?syncRunId={syncRunId}` (existing Phase 2 endpoint).
  - If `state === 'failed'`: render a banner "Your previous sync failed — Retry?" with the failure error message + a single button that POSTs to `/api/shopify/sync` (which is the same path the regular "Start sync" button uses). After the new SyncRun is created, the param drops off the URL.
  - If `state` is anything else (succeeded, running, queued): silently ignore the param and render the default onboarding view.
- **D-12:** **No auto-retry.** The button is the explicit confirmation per Phase 8 D-06. Preserves the anti-double-sync intent.

### W-2: Admin /chat banner shop resolution

- **D-13:** **`app/(embedded)/chat/page.tsx` resolves shop server-side from the embedded session token.** Use the existing `withShopifySession` pattern adapted for Server Component context (Server Components can read request headers but not invoke the App Router middleware wrapper directly — the cleanest path is to decode the session-token header in the page itself or pass shop down from a layout that does). Planner picks the exact mechanism:
  - Option a (recommended): Move `getActiveChatModel(shop)` call below a server-side `shop` resolution from the request headers via a small helper in `lib/shopify/server-resolve-shop.ts`. The helper decodes the session token from the `X-Shopify-Session-Token` request header (or `Authorization: Bearer …`) and returns the verified shop. If header is absent, fall back to `searchParams.shop` (current behavior preserved as fallback).
  - Option b: Lift `shop` from a parent layout Server Component.
- **D-14:** **Banner text contract stays byte-precise.** `Preview mode — using your real catalog · Model: {model.displayName}` with em-dash U+2014 and middle-dot U+00B7. The existing Phase 4 byte-precision tests must still pass.
