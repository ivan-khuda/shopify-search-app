## Known Gaps

### Bundle Size Regression (D-14) — OPEN

**Test:** `__tests__/bundle-build.test.ts > storefront bundle size is < 250KB minified (D-14)`
**Status:** FAILING
**Size:** 1420KB (threshold: 250KB)
**Root cause:** Plan 08.1-03 wired real `ChatPane`, `HistoryPanel`, `SavedProductsPanel` and `DbBacked` stores into `StorefrontDrawer`. This pulls the full `@/lib/chat-ui` surface and its dependencies into the storefront esbuild entry (`extensions/chat-drawer/src/entry.tsx`), bloating the bundle from 197KB (pre-08.1-03) to 1420KB.
**Fix options:**
1. Code-split: lazy-load DrawerBody panes via dynamic `import()` in entry.tsx so the storefront initial JS stays lean and ChatPane/HistoryPanel chunks are loaded on first open.
2. Cap relaxation: raise the 250KB D-14 threshold in bundle-build.test.ts with PROJECT.md rationale (DrawerBody composition is the correct V1 architecture; the cap was set pre-composition).
**Decision deferred:** The user explicitly chose to continue rather than fix Plan 08.1-03 in-place. This gap is tracked for a follow-up phase.
**Impact:** This is a performance regression, not a functional one. The storefront drawer works correctly end-to-end; it just loads a larger initial bundle. The V1 milestone audit is source-code correctness based; the bundle size gate is a delivery-quality metric tracked separately.
