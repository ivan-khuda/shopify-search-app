<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Email link (untrusted clipboard / forwarded) → `/onboarding?retry={syncRunId}` → /api/shopify/sync/status | The `syncRunId` is opaque (UUID) and the status endpoint already validates `run.shop !== ctx.shop → 403` (Phase 2 SYN-07); no new validation needed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08.1.04-01 | Tampering | `?retry={syncRunId}` URL parameter | accept | A guessed/forged syncRunId either does not exist (HTTP 404 → silent dismiss) or belongs to another shop (HTTP 403 → silent dismiss). The shop-scoping check in `/api/shopify/sync/status` is the trust gate. Even if attacker crafts a valid-format ID, no cross-shop data leaks via the banner because the lookup fails the ownership check. |
| T-08.1.04-02 | Information Disclosure | `data.errors[0]` rendered inside banner | accept | The errors come from the user's own SyncRun row (shop-scoped), so rendering them in their own admin UI is intentional. Errors are sync failure messages set by `inngest/functions/sync-products.ts` (Phase 2) — no PII outside the merchant's own data. |
| T-08.1.04-03 | Repudiation | retry button click | accept | The button reuses `handleStartSync` which posts to `/api/shopify/sync`; the existing Phase 2 SYN-04 / SYN-08 idempotency (sha256(shop+5-min-bucket)) and the audit trail in the `SyncRun` table provide non-repudiation. |
| T-08.1.04-04 | Denial of Service | repeated retry button clicks | accept | Phase 2 SYN-08 de-duplicates POST `/api/shopify/sync` within the 5-minute idempotency bucket. The retry button is already guarded by `handleStartSync`'s `if (syncing) return;` check (line 23 of current onboarding page). |
| T-08.1.04-05 | Spoofing | session-token on the GET status request | mitigate | The Bearer token is obtained via `shopify.idToken()` (App Bridge), which the Phase 1 FND-05 helper `verifyShopSessionToken` validates server-side at the status endpoint. No new auth surface in this plan. |
</threat_model>
