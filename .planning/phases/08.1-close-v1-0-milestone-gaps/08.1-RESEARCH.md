# Phase 8.1: Close v1.0 milestone gaps - Research

**Researched:** 2026-05-27
**Domain:** Next.js 16 file conventions ¬∑ Shopify App Proxy URL routing ¬∑ Server Component header access ¬∑ `lib/chat-ui` composition contract ¬∑ Polaris `s-banner` retry UX ¬∑ Vitest path-alias resolution under `extensions/`
**Confidence:** HIGH (core composition pattern is fully reverse-engineered from the existing admin shell; the one MEDIUM-confidence area is whether B-1's rename direction is correct given Next.js 16 deprecated the `middleware.ts` convention ‚Äî see Open Question O-1)

## Summary

This is a closure / hygiene phase, not a feature phase. The v1.0 milestone audit identified four BLOCKERS, three WARNINGS, and a back-mark hygiene gap that together prevent the storefront visitor chat (the V1 core-value flow) from working end-to-end despite per-phase verification passing. All seven defects are tightly localized ‚Äî most are single-line or single-file edits ‚Äî and the components, adapters, stores, and routes they touch are already built, unit-tested, and shipped. The phase's job is composition, not invention.

The non-obvious risks this research surfaces:

1. **Next.js 16.1.6 deprecated the `middleware.ts` convention and renamed it to `proxy.ts`.** The audit's B-1 finding ("Next.js auto-registers ONLY middleware.ts") contradicts the Next.js 16 file-conventions reference (verified at `https://nextjs.org/docs/app/api-reference/file-conventions/proxy`). The current `proxy.ts` with `export function proxy(...)` IS the Next.js 16 idiomatic shape. If B-1 is unverified the rename will **break** middleware registration that currently works. See O-1 below.
2. **Server Components cannot read arbitrary request headers without going through a helper that touches `next/headers`'s async `headers()` API.** W-2's "decode Bearer header in the page" path needs the Next.js 16 idiom, not the route-handler pattern from `lib/shopify/auth.ts`.
3. **Vitest's `@/` alias resolves to the project root and works under `extensions/` paths.** The existing RED test at `extensions/chat-drawer/src/components/__tests__/StorefrontDrawer.test.tsx` already imports via `@/extensions/...` and is in the include glob ‚Äî no new vitest config needed.
4. **The drawer composition has a non-obvious entry-point seam.** `entry.tsx` resolves `visitorId` from localStorage (mirroring `StorefrontAdapter`) and passes it as a prop. The StorefrontDrawer therefore receives `shop`, `visitorId`, `customerId` as props ‚Äî those are the inputs for the DB-backed store hooks. The composition is "plumb the props through, don't re-derive."

**Primary recommendation:** Treat this as a six-Plan closure phase (matches CONTEXT D-02). Reconcile O-1 with the user before any rename. Plan B (storefront bundle + adapter endpoint + test pin) is the only multi-file atomic change ‚Äî bundle them in a single commit so the test suite never sees a half-fixed adapter. Everything else is single-file, single-commit-per-fix.

|------------|-------------|----------------|-----------|

| Storefront JS bundle URL resolution | Theme App Extension JS (vanilla loader.js, storefront DOM) ‚Üí App Proxy backend (`/api/proxy/_meta/bundle-url`) | ‚Äî | Loader fetches manifest from storefront origin ‚Üí Shopify App Proxy forwards to backend ‚Üí backend returns the absolute CDN URL of the React bundle. |
| Storefront chat UI composition | Storefront client bundle (React, dynamically imported) | ‚Äî | Mounted under `<smartdiscovery-app>` custom element; uses DOM-only React, no SSR. |
| Storefront chat backend calls | Storefront client ‚Üí Shopify App Proxy ‚Üí App Proxy route handlers (`/api/proxy/chat`, `/conversations`, `/saved-products`) | ‚Äî | App Proxy paths are same-origin from the merchant's storefront and HMAC-signed by Shopify. |
| Visitor identity | Storefront client (`localStorage['smartdiscovery.visitor_id']`) + DB (`Conversation.visitorId`, `SavedProduct.visitorId`) | App Proxy routes (forward visitor_id from request body) | UUID minted on first visit; lifted into customer_id when shopper logs in (IDN-02). |

### Pattern 2: Server Component shop resolution (W-2)

**What:** Decode the embedded session-token header inside a Server Component without going through the route-handler `withShopifySession` wrapper (which expects `Request`, not the headers object).

**Example:**
```tsx
// Source: synthesized from CONTEXT D-13 + lib/shopify/auth.ts verifyShopSessionToken
// New helper at lib/shopify/server-resolve-shop.ts (or app/(embedded)/_lib/)
import { headers } from 'next/headers';
import { shopifyClient } from '@/lib/shopify/client';

export async function resolveShopFromRequest(): Promise<string | null> {
  const h = await headers();                          // Next.js 16 ‚Äî `headers()` is async
  const authHeader = h.get('authorization');
  if (!authHeader?.startsWith('Bearer ')) return null;

  const token = authHeader.slice('Bearer '.length);
  try {
    const payload = await shopifyClient.session.decodeSessionToken(token);
    if (!payload.dest) return null;
    const shop = new URL(payload.dest).hostname;
    return shop.endsWith('.myshopify.com') ? shop : null;
  } catch {
    return null;
  }
}
```

Then in `app/(embedded)/chat/page.tsx`:
```tsx
// Server Component
export default async function ChatPage({
  searchParams,
}: { searchParams: Promise<{ shop?: string }> }) {
  const { shop: shopFromQuery } = await searchParams;
  const shopFromSession = await resolveShopFromRequest();
  const shop = shopFromSession ?? shopFromQuery ?? '';

  const model = await getActiveChatModel(shop);
  // ‚Ä¶ existing banner JSX byte-precise unchanged ‚Ä¶
}
```

**Key points:**
- `headers()` returns a Promise in Next.js 15+; must `await`. [CITED: https://nextjs.org/docs/app/api-reference/functions/headers ‚Äî "headers is an async function"]
- Bearer-token decoding via `shopifyClient.session.decodeSessionToken` already in use at `lib/shopify/auth.ts:35` ‚Äî reuse the same primitive.

## Common Pitfalls

### Pitfall 1: Renaming proxy.ts ‚Üí middleware.ts on Next.js 16 may BREAK auth gating

**What goes wrong:** Next.js 16.1.6 file-conventions reference at `nextjs.org/docs/app/api-reference/file-conventions/proxy` explicitly says:

> "**Note**: The `middleware` file convention is deprecated and has been renamed to `proxy`."

And the migration codemod (`npx @next/codemod@canary middleware-to-proxy .`) does the rename in the OPPOSITE direction the audit recommends. Version history table shows: `v16.0.0  Middleware is deprecated and renamed to Proxy`.

The current `proxy.ts` with `export function proxy(...)` IS the Next.js 16 idiomatic shape. Renaming it to `middleware.ts` may register nothing (the convention is removed in v16) and would leave `/onboarding`, `/chat`, `/settings` unprotected ‚Äî which is precisely the symptom the audit reports.

**Why it happens:** The audit's B-1 narrative is from a pre-Next.js-16 mental model. The audit's verification heuristic ("anonymous direct hits load the embedded shell instead of redirecting to /api/auth") IS valid as a symptom check ‚Äî but the **cause** may not be the file name. Possible alternate causes worth verifying:
1. `proxy.ts` is registered correctly but the `matcher` only covers `/onboarding/:path*` and `/chat/:path*` and **NOT** `/settings/:path*` ‚Äî meaning `/settings` would be unprotected even with a working proxy. [VERIFIED: `proxy.ts:32` matcher = `['/onboarding/:path*', '/chat/:path*']` ‚Äî no `/settings`. CONTEXT D-04 doesn't preserve a `/settings` matcher entry either.]
2. The session-storage lookup in `proxy.ts` may fail silently if `sessionStorage.loadSession` throws inside the edge runtime (proxy can run on Node.js per v16 default, but Prisma session storage might or might not work in proxy context).
3. The redirect-to-auth path returns 302 even for valid sessions if `searchParams.get('shop')` returns null, which can happen on a direct navigation that doesn't carry the `?shop=` query.

**How to avoid:**
- **Before any rename**, the planner MUST manually verify the actual symptom in the dev server:
  ```bash
  bun dev &
  sleep 5
  curl -sI http://localhost:3000/onboarding | head -5
  # Expected if proxy.ts IS registered: HTTP/1.1 307 Temporary Redirect ‚Üí /api/auth
  # Observed if NOT registered: HTTP/1.1 200 OK (embedded shell HTML)
  ```
- **If proxy.ts IS already registering correctly**, B-1 reduces to "extend the matcher to cover `/settings`" ‚Äî not a rename. Document the actual observed cause in Plan A. Then make the minimum-viable fix; don't blindly execute the locked D-04 rename direction.
- **If proxy.ts is NOT registering** (i.e., Next.js 16 genuinely does not register the legacy `middleware.ts` AND somehow also does not register `proxy.ts`), then the rename is correct ‚Äî but the planner must explain why both conventions failed simultaneously.

**Warning signs:** Dev server starts but no log line about "Loaded proxy" / "Loaded middleware". `bun dev` showing the protected route returning 200 for an anonymous request.

**Status:** Surfaced as Open Question O-1. User must confirm direction before Plan A executes.

### Pitfall 2: Vitest @/ alias works under extensions/ but TypeScript may not

**What goes wrong:** `vitest.config.ts` resolves `@/` to the project root via `alias` config. `tsconfig.json` includes `extensions/chat-drawer/**/*.ts(x)` (via `"**/*.tsx"` in `include`) and resolves `@/*` to `./*`. So `@/lib/chat-ui` imports under `extensions/` work in BOTH vitest AND `tsc --noEmit`.

But the Theme App Extension build itself (`shopify app build` / `bundle.config.js`) may NOT honor the project-root `@/` alias. If `extensions/chat-drawer/` uses a separate bundler config (esbuild via shopify CLI), the build of `entry.tsx ‚Üí StorefrontDrawer.tsx ‚Üí @/lib/chat-ui` may fail or silently emit broken module paths.

**Why it happens:** Vitest and tsc share the project-root TypeScript paths; the Shopify CLI's extension bundler is independent.

**How to avoid:**
- Verify after Plan C composition edits: `bunx shopify app build` (or whatever the equivalent extension-build command is ‚Äî check `package.json` scripts).
- If alias resolution fails in the extension bundle, fall back to relative imports: `import { ChatPane } from '../../../../lib/chat-ui'` (or configure the extension bundler's alias map ‚Äî beyond V1 scope).

**Warning signs:** All unit tests pass, but `shopify app dev` shows a runtime error in browser console: "Cannot find module @/lib/chat-ui" or a stack trace referencing the unresolved bundle path.

**Mitigation strategy:** Plan C's verification step should include a build sanity check. Defer to relative imports if the alias is unsupported.

### Pitfall 3: `useDbBackedHistoryStore` requires a non-empty `visitorId`

**What goes wrong:** `DbBackedHistoryStore` constructor at `lib/chat-ui/stores/db-backed.ts:47-53` throws `'DbBackedHistoryStore requires a non-empty visitorId'` if `visitorId` is missing or empty. The `useMemo` in the hook (`stores/hooks.ts:52-55`) constructs the store eagerly, so an empty `visitorId` triggers the throw at render time ‚Äî not at fetch time.

The StorefrontDrawer must therefore guard against rendering the History/Saved tabs (or instantiating the hooks) until `visitorId` is non-empty.

**Why it happens:** `entry.tsx:resolveVisitorId()` always returns a non-empty UUID (either from localStorage or `crypto.randomUUID()`), so in production this is fine. But:
- **Test render** (RED test at line 13 of `StorefrontDrawer.test.tsx`: `render(<StorefrontDrawer />)`) instantiates the drawer with no props ‚Üí `visitorId === undefined` ‚Üí if the hooks are called unconditionally, the existing RED test crashes.
- **Design-mode preview** (theme editor) ‚Äî current behavior is no mount per loader.js `Shopify.designMode` check, but StorefrontDrawer's design-mode check is at FAB click time, so the drawer could render its tabs in Theme Editor and crash.

**How to avoid:**
- Plan C composition: conditionally instantiate the DB-backed hooks ONLY when `shop && visitorId` are present.
- Falling back to the placeholder text for History/Saved when props are missing preserves the existing RED test ("no history yet" etc. is no longer asserted explicitly, but the new composition test in D-10 will assert real component presence when props ARE provided).
- Test pattern: `render(<StorefrontDrawer shop="test.myshopify.com" visitorId="v1" />)` for the new D-10 assertion; existing RED tests can stay as-is (no props ‚Üí still works because conditional hook instantiation never throws).

**Warning signs:** Test failures like `Error: DbBackedHistoryStore requires a non-empty shop` thrown during render in the existing RED tests.

### Pitfall 4: Server Component `headers()` is async in Next.js 15/16

**What goes wrong:** Old code `headers().get('authorization')` returns undefined or throws in Next.js 16. The new API is `const h = await headers();`.

**How to avoid:** Always `await headers()`. Same for `cookies()` and `draftMode()`. [CITED: https://nextjs.org/docs/app/api-reference/functions/headers ‚Äî "headers is an async function"]

**Warning signs:** TypeScript error "Property 'get' does not exist on type 'Promise<ReadonlyHeaders>'".

### Pitfall 5: The retry banner can race with an active sync

**What goes wrong:** User clicks the retry deep link from an email. They land on `/onboarding?retry=run-X`. Meanwhile, they had also clicked "Start sync" in a different tab two minutes ago ‚Äî that sync is now running. The retry-banner `useEffect` fires, finds `run-X` state is `failed`, renders the banner. User clicks "Retry sync" ‚Äî which calls POST `/api/shopify/sync` ‚Äî which (per Phase 2 SYN-08) is idempotent per 5-minute bucket and gets de-duped or creates a new run.

If de-duped, the banner stays up because `syncRunId !== null` is the dismissal condition but the user's click doesn't set `syncRunId` (the POST response is consumed but the de-dupe response may not return a new ID).

**How to avoid:** D-11's "If state is anything else, silently ignore the param" rule handles this ‚Äî if the deep-link `?retry={X}` is `failed` but there's ALSO an active run, the active polling in the existing `useEffect` will overlay the banner anyway. As long as the retry banner only renders when **both** `retryRun?.state === 'failed'` **AND** `syncRunId === null`, the active-sync UI takes precedence.

**Warning signs:** Retry banner stuck onscreen after click. Two banners rendered simultaneously (active-sync progress + retry-failed).

function DrawerBody({
  activeTab,
  shop,
  visitorId,
  customerId,
}: {
  activeTab: 'chat' | 'history' | 'saved';
  shop: string;
  visitorId: string;
  customerId: string | null;
}): React.ReactElement {

  const history = useDbBackedHistoryStore({ shop, visitorId, customerId });
  const saved = useDbBackedSavedProductsStore({ shop, visitorId, customerId });

    [saved.items],
  );

        adapter={adapter}
