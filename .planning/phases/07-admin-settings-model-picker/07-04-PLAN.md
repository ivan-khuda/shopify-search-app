    Internal:
    - `CATALOG_URL = 'https://ai-gateway.vercel.sh/v1/models'` (hardcoded — Pitfall 5 + RESEARCH.md verified; no env-var injection)
    - `TTL_MS = 15 * 60 * 1000`
    - `const cache = new Map<'catalog', { data: CatalogModel[]; expiresAt: number }>()` at module top level (per RESEARCH §Pattern 2; Vercel lambda warm-instance scope is acceptable per D-03)
    - `function mapRaw(m: RawModel): CatalogModel` — converts pricing strings to $/M via `parseFloat(m.pricing?.input ?? '0') * 1_000_000`; decorates `bestFor` from `BEST_FOR[m.id] ?? 'General purpose'`; defensive null on missing pricing per Pitfall 6
    - Filter step: `json.data.filter(m => m.type === 'language')` AND ALSO filter to `BEST_FOR-keyed ids OR id === activeId` per Open Question 1 resolution. Since fetchModelCatalog has no `activeId` parameter, change to filter only to `BEST_FOR-keyed ids` — the resolver and settings page each add the active model to the displayed set if it falls outside the curated list. (Implementation note: this is the simplest split; the alternative "pass activeId in" couples concerns badly.)

    Behavior contract:
    1. On cache hit within TTL: return `{ models: cached.data, stale: false, coldStartFallback: false }` without fetching.
    2. On cache miss: `fetch(CATALOG_URL, { cache: 'no-store' })` — MUST include `cache: 'no-store'` (Pitfall 4); parse JSON; filter type==='language' AND BEST_FOR-keyed; map; populate cache with `expiresAt = Date.now() + TTL_MS`; return `{ models, stale: false, coldStartFallback: false }`.
    3. On fetch failure (network error, non-2xx status, JSON parse error) WITH a stale cache entry: return `{ models: cached.data, stale: true, coldStartFallback: false }`. Do NOT update the cache's expiresAt (this is best practice — let it reattempt on next call).
    4. On fetch failure WITHOUT a cached entry (cold-start): return the hardcoded DEFAULT_MODEL-only row from RESEARCH.md §Pattern 2 (`id: 'google/gemini-2.5-flash'`, `displayName: 'Gemini 2.5 Flash'`, `provider: 'google'`, `contextWindow: 1_000_000`, `inputPricePerMillion: 0.3`, `outputPricePerMillion: 2.5`, `bestFor: 'Fastest, low cost — great default'`) with `coldStartFallback: true`. Do NOT call into `getActiveChatModel.ts` — circular import (RESEARCH §Anti-Patterns).

    Constraints:
    - Zero `console.log` / `console.error` / `console.warn` calls anywhere in this file (CLAUDE.md hard rule); errors are silently caught and translated to fallback states.
    - No `process.env` reads — the catalog endpoint requires no auth (RESEARCH verified).
    - No `any` types — use the `RawModel` interface internally to type the JSON shape.
    - The DEFAULT_MODEL-only fallback row is the ONE place the cold-start row is materialized. Plan 06's resolver re-uses the canonical `'google/gemini-2.5-flash'` id literal from `getActiveChatModel.ts`; the cold-start row in model-catalog.ts re-emits the id as a string literal because importing from getActiveChatModel.ts would create a circular import.

    Per CONTEXT.md "Resolved Claude's-Discretion items" — this module is the planner-chosen co-located catalog client (no separate `lib/ai/` directory; no `ai-gateway-client.ts` helper). Inline `fetch()` matches the "least coupling, most direct" recommendation.
