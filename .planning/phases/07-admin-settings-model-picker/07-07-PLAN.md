Implement `app/api/settings/model/route.ts` — the PATCH endpoint the settings form posts to when a merchant clicks Save. Per CONTEXT.md "Resolved Claude's-Discretion items" the route is `/api/settings/model` (not `/api/settings`) — most specific, lowest coupling with Phase 8.

Purpose: Provide the single write path for the active-model selection. Plan 08's client form posts here on Save. The route enforces multi-tenancy (shop sourced from session, never body), defense-in-depth (catalog membership check), and CLAUDE.md hygiene (no console logs).

    - `export const PATCH = withShopifySession(async ({ shop, req }) => { ... });`

    Algorithm:
    1. Parse request JSON; on parse error return `Response.json({ error: 'invalid_body' }, { status: 400 })`.
    2. Zod validate body against `z.object({ activeChatModelId: z.string().min(1).max(200) })`. On safeParse failure return `invalid_body`.
    3. Fetch catalog via `fetchModelCatalog()` — this hits the 15-min cache 99% of the time, no measurable latency.
    4. If `!models.some(m => m.id === parsed.data.activeChatModelId)`, return `Response.json({ error: 'unknown_model_id' }, { status: 400 })` WITHOUT calling upsert.
    5. Call `shopSettingsRepository.upsert(shop, parsed.data.activeChatModelId)`.
    6. Look up the matching catalog entry's displayName: `const display = models.find(m => m.id === parsed.data.activeChatModelId)!.displayName;` (the `!` is safe — step 4 already proved membership)
    7. Return `Response.json({ ok: true, displayName: display })`.

    Constraints:
    - Multi-tenancy lock: `shop` is derived strictly from the `withShopifySession` ctx. NEVER read shop from request body or query string. Even if a tampered request includes `shop: 'evil-shop'` in the body, the Zod schema doesn't declare `shop`, and the upsert call uses the ctx.shop.
    - Zero `console.*` calls (CLAUDE.md hard rule). Errors return structured codes only.
    - No PII / no auth tokens in any response body — only `{ ok, displayName }` or `{ error: <code> }`.
    - The route ONLY supports PATCH for V1. Do NOT export GET — the Settings page (Plan 08) reads the active model via `getActiveChatModel(shop)` server-side, not via this route. (Open Question 2 resolution: keep the surface minimal; Phase 8 can add sibling routes `/api/settings/cap`, `/api/settings/email` without renaming this one.)
    - Do NOT add CSRF protection beyond the session token — the Bearer Auth path is the CSRF lock (cross-origin requests can't mint a valid session token; RESEARCH §Security).
    - Do NOT add per-request logging or analytics — Phase 8 may add observability; out of scope here.
    - Cold-start catalog state: When `fetchModelCatalog()` returns `coldStartFallback: true`, only `google/gemini-2.5-flash` is in the catalog. Any other submitted id correctly returns `unknown_model_id`. The settings UI disables Save in cold-start state (Plan 08) so the merchant cannot reach this code path; the defense-in-depth check is belt-and-suspenders.

    Error code surface (return as `{ error: <code> }` with the specified status):
    - `missing_token` (401) — emitted by withShopifySession when Authorization header missing
    - `invalid_token` (401) — emitted by withShopifySession when JWT decode fails
    - `invalid_dest` (401) — emitted by withShopifySession when payload.dest is invalid
    - `invalid_shop_domain` (401) — emitted by withShopifySession
    - `no_offline_session` (401) — emitted by withShopifySession
    - `invalid_body` (400) — emitted here when JSON parse or Zod validation fails
    - `unknown_model_id` (400) — emitted here when activeChatModelId is not in the catalog

    The 401 codes are produced by `withShopifySession` already (see lib/shopify/auth.ts); the route handler only authors the 400 codes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → PATCH endpoint | Bearer session-token (1-min lifetime); offline session is the second gate |
| Request body → DB write | Zod validation + catalog membership check are the two gates between client input and persistent state |
| Catalog fetch → membership check | Catalog may be stale or cold-start; cold-start state contains only DEFAULT_MODEL — Save is UI-disabled in this state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-07-01 | Tampering | Cross-shop write — Shop A sends a PATCH containing `shop: 'shop-b'` in the body | mitigate | The Zod schema doesn't declare `shop`; the upsert call uses ctx.shop from withShopifySession; route test asserts the multi-tenancy lock |
| T-07-07-02 | Tampering | Client submits arbitrary `activeChatModelId` (e.g., `"google/gemini-evil"`) to persist a string outside the catalog | mitigate | Catalog membership check (step 4) returns 400 before upsert; the resolver's catalog-or-synthesize fallback handles any pre-existing bad rows |
| T-07-07-03 | Tampering | Client submits a 100KB activeChatModelId to fill the DB | mitigate | Zod max(200) caps length; DB column is `String` (Postgres TEXT) but the application-layer cap is the gate |
| T-07-07-04 | Information disclosure | Route logs the session token or Authorization header in an error path | mitigate | Zero `console.*` calls; route test spies on console.log/warn/error and asserts zero calls across all branches |
| T-07-07-05 | Information disclosure | 500 error response leaks a stack trace including DB connection details | mitigate | All errors return `{ error: <code> }` only; no exception bodies surface; uncaught exceptions are caught by withShopifySession's wrapper (only ShopifyAuthError is translated; other exceptions throw and are handled by Next.js default 500 response — which is the framework default and does not include stack in production builds) |
| T-07-07-06 | CSRF | Cross-origin POST attempts to change a victim's active model | mitigate | Bearer session-token Auth path requires the App Bridge runtime to mint a token; cross-origin requests cannot do this (RESEARCH §Security) |
| T-07-07-07 | Replay | Old PATCH request replayed | accept | Session tokens are 1-min lifetime per Shopify spec; offline session can be revoked; the operation is idempotent (upsert with same payload is a no-op for state) |
| T-07-07-08 | Denial of service | Catalog endpoint is slow, causing PATCH requests to back up | mitigate | `fetchModelCatalog` hits the 15-min cache on virtually every call (saves are infrequent); cold-start fallback returns immediately with DEFAULT_MODEL-only catalog |
| T-07-07-SC | Tampering | npm/pip/cargo installs in this plan | accept | No new packages — uses existing zod 4.3.6, withShopifySession, fetchModelCatalog, repository; slopcheck N/A |
</threat_model>
