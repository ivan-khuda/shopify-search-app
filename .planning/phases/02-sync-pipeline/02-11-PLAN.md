  <how-to-verify>
    Perform the following three checks against your Shopify dev store. Each takes 2-5 minutes.

    **Check A — Inngest dev workflow runs end-to-end**
    1. Confirm `.env` has `INNGEST_DEV=1` (D-12). If missing: add the line.
    2. Confirm local Postgres is up: `docker compose ps` → postgres service `Up (healthy)`. If not: `docker compose up -d`.
    3. Reset to a clean schema: `bunx prisma migrate reset --force && bunx prisma migrate deploy`.
    4. Terminal A: `bun dev` (Next.js).
    5. Terminal B: `bunx inngest-cli@latest dev -u http://localhost:3000/api/inngest`. Wait for "Inngest dev server running at http://localhost:8288" and confirm in the dashboard's "Functions" tab that `sync-products` is registered.
    6. Install the app on your Shopify dev store (OAuth roundtrip; you've done this in Phase 1).
    7. On `/onboarding`, click "Start sync". Within 2 seconds, the button is replaced by the progress UI (progress bar + counter + state badge).
    8. In the Inngest dashboard (`localhost:8288`), watch the `shopify/product.sync` event execute step-by-step. Each step ID should match the pattern `fetch-batch-<cursor>`, `upsert-batch-<cursor>`, `persist-cursor-<cursor>`.
    9. Open `bunx prisma studio` in a new terminal. Watch the `sync_runs` table: `processedCount` increments per batch; `cursor` becomes non-null; final `state` = succeeded (or partial if your dev store has products that fail mapping).
    10. The onboarding page shows the success banner with "Open admin chat" CTA.

    **Check B — Webhook delivery + dedup**
    1. Edit `shopify.app.toml` to declare webhook subscriptions (D-16):
       ```toml
       [[webhooks.subscriptions]]
       topics = ["products/create", "products/update", "products/delete"]
       uri = "${HOST}/api/shopify/webhook"
       ```
    2. Run `bunx shopify app deploy` to register the subscriptions against your dev store.
    3. In the Shopify admin, edit a product (change its title or price). Within ~30 seconds, watch terminal A for the webhook POST — should log nothing sensitive (D-10/T-2-leak preserved) but the request completes 200.
    4. Open Prisma Studio's `webhook_events` table — one new row with a fresh `eventId`.
    5. Open the `products` table — the edited product's row has the new title/price AND a non-null `updatedAtShopify`.
    6. In the Shopify admin's "Webhooks" panel for the app, click "Send test notification" for the same event. The handler returns 200 BUT the `products` row is not modified a second time (dedup via P2002). The `webhook_events` table also has no new row (the `eventId` collision blocked it).

    **Check C — Resume on simulated timeout**
    1. Start a sync that has at least 3 batches (`processedCount` reaches ~100 in the UI).
    2. In Terminal B (Inngest CLI), press Ctrl+C to kill the worker mid-run.
    3. Restart Terminal B: `bunx inngest-cli@latest dev -u http://localhost:3000/api/inngest`.
    4. The Inngest dashboard should show the function resuming. Step IDs already completed (`fetch-batch-null`, `upsert-batch-null`, etc.) replay as memoized (instant). New steps continue from the last persisted cursor.
    5. Confirm `processedCount` in `sync_runs` continues from where it stopped (no double-counting; no restart from zero).

    Once all three checks pass, type "approved" to advance.

    If anything fails:
    - Check A failure → likely a wiring bug between sync route, Inngest function, or status route. Re-run Plans 06/07/08 gates.
    - Check B failure → either HMAC verification is wrong (re-verify D-10 in Plan 09) or the dedup is wrong (re-verify the P2002 catch).
    - Check C failure → step IDs aren't deterministic. Re-verify Plan 06's step ID interpolation.
  </how-to-verify>
