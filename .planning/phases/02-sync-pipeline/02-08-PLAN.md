Target shape (from 02-PATTERNS.md §"app/api/shopify/sync/status/route.ts"):

```typescript
import { NextResponse } from 'next/server';
import { withShopifySession } from '@/lib/shopify/auth';
import { prisma } from '@/lib/db/client';

export const GET = withShopifySession(async ({ shop, session, req }) => {
  void session;
  const { searchParams } = new URL(req.url);
  const syncRunId = searchParams.get('syncRunId');

  if (!syncRunId) {
    return NextResponse.json({ error: 'missing_sync_run_id' }, { status: 400 });
  }

  const run = await prisma.syncRun.findUnique({ where: { id: syncRunId } });
  if (!run) {
    return NextResponse.json({ error: 'sync_run_not_found' }, { status: 404 });
  }
  if (run.shop !== shop) {
    return NextResponse.json({ error: 'wrong_shop' }, { status: 403 });
  }

  return NextResponse.json({
    state: run.state,
    processedCount: run.processedCount,
    totalCount: run.totalCount,
    errors: run.errors,
    startedAt: run.startedAt,
    finishedAt: run.finishedAt,
  });
});
```

Note on order: load first, then check shop. This is intentional — checking `findFirst({ where: { id, shop } })` would also work and would never leak the existence of cross-shop runs (returns 404 instead of 403). For V1, the 403 vs 404 distinction is informative; if a security review later demands existence-hiding, switch to findFirst with both fields in the where clause.

T-2-iso mitigation: `run.shop !== shop` check is the cross-shop guard. Without it, any merchant with a valid session token could poll any sync run by guessing IDs.
