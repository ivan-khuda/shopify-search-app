Function shape:

```typescript
export const syncProductsFunction = inngest.createFunction(
  { id: 'sync-products', retries: 3 },
  { event: 'shopify/product.sync' },
  async ({ event, step }) => {
    const { syncRunId, shop } = event.data as { syncRunId: string; shop: string };
    // 1. load offline session for shop
    // 2. step.run('mark-running') — transition queued→running, set startedAt
    // 3. step.run('fetch-total-count') — call fetchTotalCount; write SyncRun.totalCount
    // 4. loop while hasNextPage (or first iteration):
    //    step.run(`fetch-batch-${cursor || 'start'}`)    — call fetchProductBatch
    //    step.run(`upsert-batch-${cursor || 'start'}`)   — for each product: try upsert, catch into batchErrors; if all failed → throw
    //    step.run(`persist-cursor-${cursor || 'start'}`) — prisma.syncRun.update { cursor: endCursor, processedCount: { increment }, errors: { push } }
    //    cursor = endCursor
    // 5. step.run('finalize') — set state to succeeded (zero errors) | partial (some errors) | leave to retry on throw
  }
);
```

State transitions (from D-03, D-15):
- queued → running: at function start (mark-running step).
- running → succeeded: in finalize step when SyncRun.errors.length === 0.
- running → partial: in finalize step when SyncRun.errors.length > 0.
- running → failed: handled by Inngest's `onFailure` hook (or implicit when retries exhaust) — write `state = 'failed'`, `finishedAt = new Date()`.

Step ID determinism (D-01 locked): never use Date.now(), random, or anything non-deterministic in step IDs. Cursor is the only allowed dynamic component, with `'start'` for the first iteration.

Error policy (D-15 locked):
- Per-product try/catch around `productRepository.upsertProduct(shop, mapToUpsertInput(node))`.
- `batchErrors[]` accumulates `{shopifyId, message}` per failed product.
- After the per-product loop: if `batchErrors.length === batch.products.length` (all failed) → throw → Inngest retries the step.
- Otherwise: return the partial errors; the persist-cursor step pushes them into `SyncRun.errors[]` (each entry is `JSON.stringify({shopifyId, message})` since the column is `String[]`).

       a. `const batch = await step.run(\`fetch-batch-${cursor ?? 'start'}\`, async () => fetchProductBatch(session, cursor, 100));` — D-02 locks batch size to 100. Returns `{products, endCursor, hasNextPage}`.

       b. `const { errors: upsertErrors } = await step.run(\`upsert-batch-${cursor ?? 'start'}\`, async () => {`
          `  const batchErrors: Array<{ shopifyId: string; message: string }> = [];`
          `  for (const node of batch.products) {`
          `    try { await productRepository.upsertProduct(shop, mapToUpsertInput(node)); }`
          `    catch (err) { batchErrors.push({ shopifyId: node.id, message: err instanceof Error ? err.message : String(err) }); }`
          `  }`
          `  if (batch.products.length > 0 && batchErrors.length === batch.products.length) {`
          `    throw new Error(\`Full batch failed: \${batchErrors.map(e => e.message).join(', ')}\`);`
          `  }`
          `  return { errors: batchErrors };`
          `});`

       c. `await step.run(\`persist-cursor-${cursor ?? 'start'}\`, async () => {`
          `  await prisma.syncRun.update({`
          `    where: { id: syncRunId },`

          `      cursor: batch.endCursor,`
          `      processedCount: { increment: batch.products.length - upsertErrors.length },`

          `  });`

This guarantees that when all retries are exhausted the SyncRun row reflects state='failed'.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Inngest event → step function | Event payload is trusted to come from our own POST `/api/shopify/sync` (Inngest signing key verifies origin in prod); function still re-derives shop from the event data + loads the offline session before any GraphQL call. |
| step.run boundary | Each step's return value is memoized by Inngest. The function MUST treat any cross-step state as flowing through step return values, not closure variables — closures are not restored on replay. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-error | Reliability (batch failure cascading) | `upsert-batch` step | mitigate | Per-product try/catch (D-15); successful rows complete, failures accumulate into `batchErrors[]`; throw only when EVERY product in a batch fails (so Inngest retries the batch). On any partial-batch outcome, errors flow to `SyncRun.errors[]`; final state `partial` reflects the situation. Verified by Plan 01 RED tests for SYN-03. |
| T-2-resumable | Availability (lost work on cold restart) | Entire sync function | mitigate | Deterministic step IDs `fetch-batch-${cursor}`, `upsert-batch-${cursor}`, `persist-cursor-${cursor}` (D-01); Inngest memoizes by step ID, so a replay after Vercel timeout skips completed steps and resumes from the last unfinished one. The persisted `SyncRun.cursor` is the durable resume anchor. Verified by Check C in Plan 11 (manual smoke: kill `inngest-cli` mid-run, restart, confirm processedCount continues). |
| T-2-leak | Information Disclosure (token in event payload) | `inngest.send` call site | mitigate | Event `data` payload contains ONLY `{ syncRunId, shop }` — never `accessToken`. The function reloads the offline session via `sessionStorage.loadSession(shopifyClient.session.getOfflineId(shop))` inside the first step. Grep gate in Plan 11 confirms no `accessToken` reference in the sync POST route's `inngest.send` call. |
</threat_model>
