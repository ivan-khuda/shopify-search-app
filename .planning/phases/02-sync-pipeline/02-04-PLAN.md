    Plan 02 authored a migration SQL file and schema changes. Neither has been applied to the running database. This task applies them and regenerates the Prisma client.
  </what-built>
  <how-to-verify>
    Run these commands in order (you, the developer, run them — Claude cannot reach an interactive TTY for `migrate dev`):

    1. Confirm Postgres is up: `docker compose ps` should show the postgres service `Up (healthy)`. If not: `docker compose up -d`.

    2. Confirm migration file from Plan 02 exists:
       `ls prisma/migrations/*_add_sync_pipeline/migration.sql`
       (should print a path; if not, return to Plan 02.)

    3. Apply the migration:
       `bunx prisma migrate dev --name add_sync_pipeline`
       Expected output: "Applying migration `<timestamp>_add_sync_pipeline`" followed by "Your database is now in sync with your schema." Prisma will then run `prisma generate` automatically.

       Non-TTY fallback (e.g. running in CI): `bunx prisma migrate deploy`.

    4. Verify migration status:
       `bunx prisma migrate status`
       Expected: `Database schema is up to date!` with the new `add_sync_pipeline` migration listed as applied.

    5. Verify generated types include the new models:
       `grep -c "SyncRun\\|WebhookEvent\\|SyncState" app/generated/prisma/index.d.ts`
       Expected: a count &gt; 0 (typically &gt; 5).

    6. Verify Phase 1 tables intact (no data lost):
       `bunx prisma db execute --stdin <<&lt; '\\dt'` (or use Prisma Studio at `bunx prisma studio` and visually confirm `products`, `product_variants`, `product_images`, `product_options`, `product_embeddings`, `shopify_sessions` all still exist alongside the new `sync_runs` and `webhook_events`).

    7. Confirm `app/generated/prisma/` was regenerated by checking the file mtime is recent:
       `stat -f %Sm app/generated/prisma/index.d.ts` (macOS) or `stat -c %y app/generated/prisma/index.d.ts` (Linux).

    8. Run `bunx tsc --noEmit` — should still pass; the new generated types are now available.

    Once all 8 checks pass, type "approved" and Claude will mark the plan complete and proceed to Wave 3.

    If anything fails:
    - `prisma migrate dev` errors typically mean either Postgres is unreachable (check `docker compose ps`) or the migration SQL has a syntax issue (return to Plan 02 to fix).
    - Type-check failures after `prisma generate` mean the schema doesn't match expectations (return to Plan 02).
  </how-to-verify>
